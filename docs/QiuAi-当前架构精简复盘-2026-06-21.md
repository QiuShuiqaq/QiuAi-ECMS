# QiuAi 当前架构精简复盘

## 目的

这份文档只描述当前这版真实在用的桌面端到服务端链路，帮助后续继续开发桌面端时不再混淆旧方案、浏览器兼容路径和已废弃思路。

## 当前系统的核心定位

当前 `QiuAi` 已经不是“本地直接调三方模型 API 的个人工具”。

当前真实定位是：

- 桌面端负责产品工作台、草稿编辑、任务发起、结果查看、导出。
- 主进程负责本地状态管理、IPC、远程授权、远程生成任务编排。
- 服务端 `qiu-commerce-license-platform` 负责授权、套餐、钱包、算力档位、生成任务队列、上游模型调用。
- 最终生成结果优先由桌面端直接下载成品，不要求所有结果都经过主站二次回传。

## 当前真实主链路

### 1. 桌面渲染层

入口主要在 [App.vue](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/renderer/src/App.vue)。

它现在主要负责：

- 维护当前菜单、当前草稿、当前结果、当前导出项。
- 加载授权状态、工作台快照、提示词模板、购买中心数据。
- 把用户操作转成 `desktopBridge` 调用。
- 根据授权状态决定是否展示 `ActivationGate` 或工作台页面。

### 2. 桌面桥接层

桥接层在 [desktopBridge.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/renderer/src/services/desktopBridge.js)。

它的职责是：

- 渲染层不直接碰 `ipcRenderer`。
- 所有桌面能力通过统一桥接函数暴露。
- 浏览器 fallback 只作为非桌面调试兼容，不是正式生产主路径。

这意味着后续你改桌面端时，优先改桥接函数和对应 IPC，不要在页面里绕过桥接直接拼底层调用。

### 3. IPC 层

共享通道定义在 [ipcChannels.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/shared/ipcChannels.js)。

当前高价值通道分两组：

- 授权/购买相关：`license:*`、`compute-package:*`、`recharge:*`
- 工作台相关：`studio:*`

主进程注册入口在 [registerIpc.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/main/src/bootstrap/registerIpc.js)。

这里把几个核心服务装配到一起：

- `authorizationService`
- `qiuAiLicensePlatformClientService`
- `cloudGenerationService`
- `studioWorkspaceService`

### 4. 主进程服务层

#### 授权与购买

[licenseIpc.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/main/src/ipc/licenseIpc.js) 当前已经是“远程授权优先”的实现：

- 激活时使用设备指纹 `deviceCode` 作为 `deviceFingerprint`
- 成功激活后把 `sessionToken`、`lastUserId`、`lastLicenseId` 写入本地设置
- 没有远程会话时，购买相关能力直接失败，错误码是 `REMOTE_AUTH_REQUIRED`

[authorizationService.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/main/src/services/authorizationService.js) 的职责是：

- 调服务端 `/api/activation/status`
- 拉取远程 `remoteServiceCapacity`
- 将远程授权状态标准化为桌面端统一结构

#### 工作台与任务

工作台主服务是 [studioWorkspaceService.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/main/src/services/studioWorkspaceService.js)。

它现在是桌面业务的中枢，负责：

- 草稿保存
- 项目管理
- 任务创建
- 运行状态快照
- 结果整理
- 导出项组织

当前生成任务不是渲染层直接打服务端，而是：

1. 渲染层触发 `createStudioTask`
2. IPC 进入 `studioWorkspaceService`
3. `studioWorkspaceService` 调用 `cloudGenerationService`
4. `cloudGenerationService` 调用远程平台 API

#### 远程生成

远程生成主逻辑在 [cloudGenerationService.js](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/main/src/services/cloudGenerationService.js)。

当前三类生成都已经走服务端：

- 文本
- 套图
- 视频

当前重要约束：

- 图片生成只接受当前保留入口 `series-generate`
- 旧的本地图片生成隐藏兼容路径已经不再作为正式业务入口
- 未登录远程授权时，文本/图片/视频生成都会 fail closed，不会偷偷回退本地旧逻辑

## 服务端当前真实职责

服务端项目是 `qiu-commerce-license-platform`。

当前对桌面端最关键的 API 有：

- `/api/activation/status`
- `/api/activation/activate`
- `/api/wallet/summary`
- `/api/service-capacity/profile`
- `/api/packages`
- `/api/orders`
- `/api/compute-packages`
- `/api/compute-package-orders`
- `/api/recharge/orders`
- `/api/generation/jobs`
- `/api/generation/jobs/[id]`
- `/api/generation/artifacts/[id]/download`

生成创建入口在 [route.ts](/abs/path/f:/Workspace_VS/QiuAi-ECMS/qiu-commerce-license-platform/src/app/api/generation/jobs/route.ts)。

生成查询入口在 [route.ts](/abs/path/f:/Workspace_VS/QiuAi-ECMS/qiu-commerce-license-platform/src/app/api/generation/jobs/[id]/route.ts)。

服务端当前已经负责：

- 校验桌面端 `sessionToken`
- 根据用户套餐和服务计划生成 `remoteServiceCapacity`
- 建立生成任务与工作项
- 控制任务轮询建议 `pollingAdvice`
- 返回成品 `artifact`

## 当前最容易混淆的 4 个点

### 1. 渲染层 1.5 秒轮询，不等于服务端 1.5 秒轮询

[App.vue](/abs/path/f:/Workspace_VS/QiuAi-ECMS/QiuAi/renderer/src/App.vue) 里现在有 `1500ms` 轮询。

这轮询的是：

- `getStudioRuntimeSnapshot()`

也就是桌面本地主进程里的运行态快照，不是直接查服务端生成状态。

所以这个 1.5 秒只是在桌面内刷新 UI 状态，不是对外部平台的 1.5 秒高频压测。

### 2. 真正访问服务端生成状态的是主进程

真正查远程生成任务状态的是 `cloudGenerationService`。

当前策略是：

- 默认回退轮询间隔 `10s`
- 如果服务端返回 `pollingAdvice`，就按服务端建议走
- 当前实际建议大致是：文本约 `4s`，图片约 `5s`，视频约 `12s`

所以你之前担心的“客户端每 3 秒查一次服务端”不是当前真实情况。

### 3. `remoteServiceCapacity` 是桌面端和服务端的关键契约

这个结构同时影响：

- 页面上的服务档位展示
- 图片并发校验
- 视频并发校验
- 文本并发上限
- 任务提交时的 `requestedConcurrency`

当前桌面端消费的关键字段有：

- `serviceTier`
- `effectiveImageConcurrency`
- `effectiveVideoConcurrency`
- `effectiveTextConcurrency`

这条契约一旦改名或改结构，桌面端生成约束会立刻失真。

### 4. 结果下载现在已经支持“优先直连”

这轮已完成对齐：

- 服务端 `artifact` 可以带 `downloadUrl`
- 桌面端下载结果时优先使用 `downloadUrl`
- 没有可用直链时，才回退到平台下载接口

这符合你后面要做的目标：

- 任务由服务端统一调度
- 结果尽量直接给客户端
- 减少主站做大文件中转的压力

## 当前业务页面的真实分工

### ActivationGate

负责：

- 显示设备码
- 执行远程激活
- 显示授权状态、激活时间、到期时间、下一步动作

不负责：

- 生成任务
- 本地 license 校验
- 本地离线授权模式

### PurchaseCenterPage

负责：

- 软件授权套餐购买
- 算力包购买
- 余额充值

它现在依赖远程 `sessionToken`，不再是独立本地商城逻辑。

### GeneratorStudioPage

负责：

- 套图生成和视频生成页面
- 远程服务档位提示
- 输入草稿编辑
- 展示当前活动任务和结果

它不直接调服务端，只通过 `App.vue -> desktopBridge -> IPC` 走主进程。

### ProductWorkbench

负责：

- 更像“项目级工作台”
- 组织产品资料、草稿、运行结果和导出

这是后面你继续做“桌面端精细化开发”的重点区域。

## 当前本地持久化分层

主进程使用 `electron-store`，当前至少有三类本地存储：

- `qiuai-settings`
- `qiuai-prompts`
- `qiuai-studio`

大致分工是：

- settings：授权平台配置、远程会话、服务容量快照
- prompts：提示词模板
- studio：工作台草稿、项目、任务、结果、运行态

所以后面如果你看到“界面状态和服务端不一致”，先分清楚是：

- 本地 store 里的旧状态问题
- 主进程快照生成问题
- 还是服务端返回结构变了

## 当前设计的稳定边界

后续继续开发时，下面这些边界尽量不要再打乱：

1. 渲染层不直接访问服务端，只访问 `desktopBridge`
2. 主进程统一管理远程授权和远程生成
3. 服务端统一决定授权、算力档位、队列、轮询建议
4. 生成结果优先直连下载，平台下载接口只做回退
5. 图片正式入口以 `series-generate` 为准，不再恢复旧本地多分支入口

## 当前仍然存在的现实问题

这版已经能跑通，但还没有完全“结构干净”。

目前仍然值得继续清理的方向有：

- 渲染层还有少量历史兼容状态和无用包装计算
- `desktopBridge` 里还有浏览器 fallback 兼容代码
- 工作台页面里还有一些旧文案和历史任务概念残留
- 桌面端菜单和页面入口仍有部分“保留但价值不高”的区域

这些不一定会立刻导致功能错误，但会持续提高你后续改桌面端的混乱度。

## 后续开发建议

如果你接下来继续精细化开发桌面端，推荐顺序是：

1. 先围绕当前这份结构，清掉重复业务入口和旧兼容路径
2. 再收敛桌面端菜单，只保留真实售卖场景必须的页面
3. 再调整客户端套餐能力、并发档位、任务展示方式
4. 最后再做桌面端和服务端联调验收

原因很简单：

- 现在服务端主链路已经成型
- 你真正不稳定的地方主要在桌面端结构和入口收敛
- 如果桌面端继续一边开发一边保留旧入口，后面会越来越难判断哪条链路才是正式链路
